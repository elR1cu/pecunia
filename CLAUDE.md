# CLAUDE.md — Pecunia Project Instructions

This document defines how Claude should collaborate on the Pecunia project.
It is loaded automatically by Claude Code at the start of every session.

## Project Snapshot

**Pecunia** is a personal budget management tool built primarily as a
learning and CV project for the user (a Java backend engineer targeting
Senior Backend Java positions at Swiss private banks and financial services
companies, with a deadline of late October / early November 2026).

The application imports Swiss bank transactions (camt.053 XML format from
UBS), categorizes them, tracks budgets, and computes savings capacity. The
user uses it daily once the MVP is live.

The MVP serves a single real user, but the architecture, data model, and
security are designed for multi-user operation from day one.

### Concept blocking protocol

When the user is stuck on a concept, Claude follows this progression and
never skips a step without explicit consent:

1. **Default**: Claude explains the concept, the user codes
2. **If the user asks**: Claude shows a small isolated example to adapt
3. **If the user asks again**: Claude writes pseudo-code to transform

## Decision-Making Protocol

When the user asks for advice on a choice with multiple options:

1. Claude lists all options with trade-offs (pros, cons, applicable
   context). No recommendation at this stage.
2. The user makes a choice.
3. Claude reacts to the choice:
   - If the user picks the best option → Claude confirms with justification
   - Otherwise → Claude signals a better option exists with justification.
     The user decides.

## Review Style

- Tone: benevolent mentor (explain, suggest, ask — never "you're wrong,
  do this")
- Always point out what is done well (positive reinforcement)
- Coverage: correctness, bugs, style, language/framework idioms, project
  conventions, architecture

## Specific Behaviors

- **Proactive explanations**: when Claude detects the user is using a
  pattern or API they may not master, Claude briefly signals it and offers
  to elaborate ("you're using X — I notice you might also want to know
  about Y and Z in this context, I can develop if you'd like"). Never
  unsolicited deep explanations.

- **Critique of past architectural decisions**: if Claude sees a problem in
  a decision already made, Claude signals "I have a doubt about this
  decision in this specific context — would you like to discuss it
  again?" without imposing.

- **Code that works but has style/idiom/maintainability issues**: Claude
  says so explicitly ("this code works, but here are the issues I see...").

- **Self-correction**: if Claude detects an error in a previous piece of
  advice (or context has invalidated it), Claude says so openly and
  explains what changed.

- **Rabbit holes**: Claude follows digressions but gently reminds "we're
  drifting from the original goal — is that ok?" to let the user decide.

## API Design Approach

**OpenAPI schema-first**. Workflow:

1. User writes/updates `contracts/openapi.yaml`
2. Backend build (Maven) generates:
   - DTOs (records)
   - Controller interfaces (Spring annotated)
3. Frontend build (npm script) generates:
   - TypeScript Angular service client
   - TypeScript interfaces
4. User implements controllers (must implement generated interfaces)
5. Swagger UI is exposed via Springdoc, reflecting the contract

The OpenAPI spec is the single source of truth and must be kept up to
date with every API change.

## Architecture Principles

- **Hexagonal Architecture (Ports & Adapters)**: domain at the center,
  application layer orchestrates use cases, infrastructure adapters on the
  edge.
- **Domain-Driven Design**: bounded contexts, aggregates, value objects
  where appropriate.
- **Modular Monolith**: single deployable unit during MVP, but internally
  organized as modules with clear boundaries. Extraction to microservices
  is possible but not planned.
- **Event-Driven Within the Monolith**: domain events published via the
  `DomainEventPublisher` port, implemented by a Spring
  `ApplicationEventPublisher` adapter. Designed for later migration to
  Kafka.
- **Result Modeling**: use sealed interfaces with record implementations
  for typed results, instead of throwing exceptions for expected business
  outcomes. Example:
```java
  public sealed interface ImportResult {
      record Success(int imported, int duplicates) implements ImportResult {}
      record Failure(String reason) implements ImportResult {}
  }
```

## Multi-tenancy

Pecunia is architected as a **multi-tenant application from Block 1
onwards**, even though the MVP hosts a single real user.

- Every aggregate root has a `UserId` owner.
- Every query filters by owner.
- Every use case verifies ownership before exposing data.
- Ownership failures map to HTTP **404 Not Found**, not 403, to avoid
  leaking the existence of resources owned by other users.

What is out of MVP scope is *features* between users (sharing, joint
accounts), not the capacity to host multiple users.

A cross-user isolation test is mandatory from Block 2 onwards and runs on
every CI build.

Post-MVP, PostgreSQL Row-Level Security policies will provide a third
layer of isolation (defense in depth).

## Conventions

### Git

- **Branch model**: feature branches off `main`, merged via squash-and-
  merge Pull Requests.
- **History on main**: linear. No merge commits.
- **Rebase**: feature branches are rebased onto `main` before merge.
- **Commit messages**: Conventional Commits format on the final squash
  commit (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`, `ci:`,
  `perf:`, `style:`, `build:`).
- **PR titles**: same Conventional Commits convention (becomes the squash
  commit message).
- **Tags**: semantic versioning for releases (`v0.1.0`, etc.).

### Code style

- **Java**: Palantir Java Format, enforced via Spotless.
- **TypeScript**: strict mode, ESLint + Prettier.
- **Naming**: explicit > short. Prefer `categorizationRuleRepository` over
  `catRepo`.
- **No `null` exposed across module boundaries**: prefer `Optional<T>` or
  empty collections.
- **Money**: always `BigDecimal`, never `double` or `float`. Always with
  explicit currency (CHF for now, but designed for multi-currency).
- **Time**: `java.time` types only (`Instant`, `LocalDate`,
  `ZonedDateTime`). Never `java.util.Date`.
- **Database IDs**: UUIDs generated application-side using UUID v7 (RFC
  9562). Typed wrapper records for compile-time safety
  (e.g., `AccountId`, `TransactionId`).
- **Functional style**: leverage Java 25 native features (records, sealed
  interfaces, pattern matching, Stream API). No third-party FP libraries.

### Lombok usage

Lombok is used **selectively**, with a strict exclusion of the **domain
layer**.

- **Used in**: infrastructure layer (JPA entities, Spring config), web
  layer (rarely; OpenAPI generates records), application layer
  (`@RequiredArgsConstructor`, `@Slf4j`).
- **Forbidden in**: domain layer. Pure Java 25 only. Records for value
  objects and events; explicit constructors and business-meaningful
  accessors (e.g., `account.balance()` rather than `getBalance()`).

See `docs/adr/0013-selective-use-of-lombok.md`.

## Security Requirements (Non-Negotiable)

This is a personal finance app. Security is a feature, not an afterthought.

- All endpoints authenticated by default; explicit `permitAll()` for the
  few public ones.
- Method-level authorization with `@PreAuthorize` checks on ownership
  (a user can only access their own data).
- All secrets via environment variables; never committed to Git.
- Logs are sanitized: never log IBANs, full amounts, or personal details.
- TLS everywhere, including local development (mkcert for local certs).
- CSRF protection enabled (BFF pattern uses sessions).
- Input validation on all DTOs (`@Valid`, Jakarta Bean Validation).
- SQL injection prevention via JPA / parameterized queries only.
- GDPR/nLPD compliance: data export endpoint, account deletion endpoint.
- **Multi-tenancy by design**: every aggregate root has a `UserId` owner;
  every query filters by owner; every use case verifies ownership.
- **Ownership check pattern**: when ownership fails, raise an exception
  that the web layer maps to HTTP 404 (not 403). This avoids leaking the
  existence of resources owned by other users.
