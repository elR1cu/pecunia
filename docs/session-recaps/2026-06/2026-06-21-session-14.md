## Key decisions made
- **Scope of this CI = build/test + PR-title lint, no SonarCloud** — Sonar needs
  an external account + `SONAR_TOKEN`, so it gets its own session to avoid a red
  CI waiting on external config.
- **Both jobs always run (no path filtering)** — the repo is small, solo, with no
  branch protection yet; path filters add complexity and the future
  "required-check-that-never-runs stays pending" footgun. Also the shared
  `contracts/openapi.yaml` feeds both sides, so naive per-folder filters would be
  wrong.
- **`pull_request` (not `pull_request_target`) for the title lint** — safer, and
  there are no forks on a personal repo.
- **Merge strategy for the 57-commit branch: rebase-merge, not squash** (option B
  or, ideally, D = light `rebase -i` curation first) — preserves a
  portfolio-worthy granular history instead of collapsing all of Block 1 into one
  opaque commit. Opening the PR locks nothing; the method is chosen at merge time.

## Concepts explained / learned
- **The frontend CI job needs a JDK**: `npm run build` → `prebuild` →
  `generate:api` invokes `7.22.0.jar` via the npm wrapper (`java -jar …` visible
  in the log). Without `setup-java`, the frontend job would fail.
- **GitHub action major versions verified before pinning** (bleeding-edge stack):
  `checkout@v5`, `setup-java@v5`, `setup-node@v5`,
  `action-semantic-pull-request@v6` are the current majors as of June 2026.
- **Opening a PR does not commit to a merge method** — squash vs rebase vs merge
  is chosen at the merge button, so the 57-commit decision can be deferred.
- **CLAUDE.md's squash convention was written for short feature branches**, not a
  whole-block integration branch; the nuance is now a memory and a pending
  CLAUDE.md edit.
