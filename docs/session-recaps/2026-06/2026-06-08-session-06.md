## Key decisions made

- **MapStruct is the project's mapper convention.** All future mappers
  use it. Establishes uniformity over local optimality.
- **Mapper method naming: `toDto` / `toEntity` / `toDomain`.** Class
  name carries the destination type; method name carries the role.
- **`ReportingPolicy.ERROR` on every `@Mapper`.** Adding a target field
  without the matching `@Mapping` fails the build. Fail-fast convention
  for the project.
- **Assertive over defensive null-handling.** Pattern matching with
  `throw IllegalStateException` for invariants the SecurityFilterChain
  guarantees. No `Optional` chains, no `@SuppressWarnings`, no recovery
  for the impossible.
- **`emailVerified` exposed in the DTO with `false` fallback.** Frontend
  decides UX (badge, banner, redirect to Keycloak verification flow).
  Block 1 keeps it simple, post-MVP can add explicit handling.
- **Tests: `@WebMvcTest` slice + `oidcLogin()` + real mapper via
  `@Import`.** Defines the project's web-layer test convention. Real
  `CurrentUserMapperImpl` is loaded, not mocked.
- **Test method naming: camelCase + `@DisplayName`** for readability.
  Avoids snake_case method names while still giving humans a readable
  description in reports.
- **Lombok stays selective per ADR-0013.** Added to pom because the
  controller layer is explicitly allowed.
- **`lombok-mapstruct-binding` deferred** until a Lombok-annotated class
  actually flows through MapStruct. YAGNI applied.

## Concepts explained / learned

- **`Authentication` vs `OidcUser`**: `Authentication` wraps the
  principal + authorities + auth state; the OIDC-specific data lives
  in `.getPrincipal()`. Casting the wrapper directly was the original
  bug.
- **Defensive vs. assertive coding**: defensive code recovers from a
  bad state (returns 401, falls back). Assertive code declares an
  invariant and throws if violated. For invariants guaranteed by an
  upstream layer (like the `SecurityFilterChain`), assertive is the
  right tool — defensive code hides bugs.
- **`Optional` is not a null-safety hammer**: it models *legitimate
  absence* in the domain. Wrapping invariants in
  `Optional.ofNullable(...)` is the same anti-pattern as `if (x == null)
  return 401;`, just in functional clothing.
- **Java 25 functional style**: switch expressions, pattern matching
  for instanceof, and records are the idiomatic Java functional
  primitives. `Optional` chains aren't always "more functional" —
  they're often less type-safe and less readable.
- **`Optional.orElse(...)` is eager, `.orElseGet(...)` is lazy.**
  Use `.orElse(constant)` for literals, `.orElseGet(() -> ...)` for
  computed defaults.
- **IntelliJ flow analysis**: `Objects.requireNonNull(...)` propagates
  non-null on the returned value, but a cast `(T) someNonNull` does not
  always propagate type narrowness. Pattern matching with `instanceof`
  is recognized natively and gives the cleanest warning-free path.
- **MapStruct gotchas**:
  - `@Mapping(expression = "java(...)")` is the escape hatch for
    multi-source field computation; verbose but the alternatives
    (`@AfterMapping` mutation, ambiguous `source = "<paramName>"`) are
    worse here.
  - Generated impl uses setters on a no-args constructor, not the
    required-args constructor. Implications for Jakarta Validation
    `@NotNull` (not enforced at construction).
- **MapStruct + Lombok ordering**: in `annotationProcessorPaths`, Lombok
  must come before MapStruct so its generated accessors are visible.
  When both processors coexist with `lombok-mapstruct-binding` being
  the canonical glue once Lombok-annotated classes are mapped.
- **Spring Boot 4 test autoconfig split**: the same per-feature module
  pattern as runtime autoconfig (already in memory). The
  `@WebMvcTest` annotation moved from `spring-boot-test-autoconfigure`
  to `spring-boot-webmvc-test-autoconfigure` (per the
  `spring-boot-starter-webmvc-test` starter). Test slices each have a
  dedicated starter to pull.
- **`oidcLogin()` mechanics**: builds an `OAuth2AuthenticationToken`
  with a `DefaultOidcUser` principal from the configured claims.
  Bypasses the actual OAuth2 login flow but populates the
  `SecurityContext` as if it had happened.
- **`StandardClaimNames`**: canonical OIDC claim name constants
  (`PREFERRED_USERNAME`, `EMAIL`, etc.). Use these instead of string
  literals to avoid typos and to express intent.

## Post-session follow-up

- **Mockito self-attach warning silenced**: wired `maven-surefire-plugin`
  with `<argLine>-javaagent:${org.mockito:mockito-core:jar}</argLine>`.
  The jar path is exposed by the `maven-dependency-plugin` `properties`
  goal (`resolve-dependency-paths` execution). Mockito now loads as a
  Java agent at JVM start instead of self-attaching dynamically — fixes
  the runtime warning and future-proofs against the JDK change that will
  forbid dynamic agent attach. Committed as `build(api): silence Mockito
  self-attach warning and harden /me slice`.
- **`MeControllerTest` hermeticity**: re-added `@MockitoBean
  ClientRegistrationRepository` to short-circuit
  `OAuth2ClientAutoConfiguration` from performing OIDC discovery against
  Keycloak at context load. Reverses the "unnecessary mock removed"
  decision noted above — it *is* necessary for the slice to be CI-safe
  (no Keycloak available there) and to stay hermetic in general. Same
  commit as the Mockito fix.
- **IntelliJ DevTools hot reload working**: unchecked **Settings →
  Build, Execution, Deployment → Build Tools → Maven → Runner →
  "Delegate IDE build/run actions to Maven"**. IntelliJ now uses its
  native incremental compiler instead of Maven; `automake` can then
  regenerate `target/classes` while the app is running, which is the
  classpath change DevTools watches for. This finally closes the
  hot-reload TODO from Session 03.
  - **Side effect**: the `spring-boot:build-info` Maven goal no longer
    runs on every IDE build. `target/classes/META-INF/build-info.properties`
    is regenerated only when Maven runs (e.g. `mvn package`,
    `mvn spring-boot:build-info`, or a CLI `mvn test`). `/actuator/info`
    will keep serving the last-generated values until then. Acceptable
    trade-off in dev; CI builds run Maven so production artifacts are
    always fresh.
