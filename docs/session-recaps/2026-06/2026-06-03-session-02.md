## Key decisions made

- **Strict env-var sourcing for the BFF client secret on the Spring side** (Option A): no fallback default in `application.yml`. Production cannot accidentally boot with a repo-known value; dev must provide a `.env` or set the env var.
- **Hybrid treatment of secrets in the realm JSON**: `testuser` password committed as an Argon2id hash (Keycloak-standard import format); `pecunia-bff` client secret kept inline as a self-described dev fixture, on the basis that it has no value outside the local Keycloak and that Keycloak intentionally does not support env-var substitution in realm import (issue [#20199](https://github.com/keycloak/keycloak/issues/20199), closed "not planned").
- **`.env` location at `apps/api/.env`**: scoped to the only actual consumer (the Spring Boot app); Docker Compose does not need it because the Keycloak side reads the secret from the realm JSON inline.
- **Postgres 18 / Keycloak 26.6 / Redis 8** rather than the latest patch in older minor branches: project is starting from scratch, supported branches make more sense.

## Concepts explained / learned

- **`spring.session.store-type` was removed in Spring Boot 3.0** (M4). User correctly pushed back when the assistant proposed it. Verified via release notes. The replacement mechanism is classpath detection with a fixed priority order across multiple session implementations; manual override is now done by defining a `SessionRepository` bean.
- **Keycloak realm import does not perform env-var substitution** by design. Issue #20199 was closed "not planned". Workarounds are wrapping the start command with `envsubst`/`sed`, using `keycloak-config-cli`, or accepting that committed realm JSONs are dev fixtures.
- **Keycloak 26.6 hashes passwords with Argon2id** by default (algorithm `argon2`, type `id`, version 1.3, memory 7168 KB, parallelism 1, hash length 32, iterations 5). The realm import accepts the hashed credential format using `secretData` and `credentialData` JSON-encoded strings.
- **Postgres 18 volume layout breaking change**: the official Docker image now expects the mount at `/var/lib/postgresql` so it can manage version-specific subdirectories internally, enabling `pg_upgrade --link` across major versions.
- **`KEYCLOAK_ADMIN`/`KEYCLOAK_ADMIN_PASSWORD` are deprecated** in favour of `KC_BOOTSTRAP_ADMIN_USERNAME`/`KC_BOOTSTRAP_ADMIN_PASSWORD`. The new variables only act on the very first startup (when no admin user exists in the database).
- **`.env.example` is a convention, not a mechanism**: it documents expected variables, enables one-line onboarding (`cp .env.example .env`), and acts as a drift detector against `application.yml` references.

## Issues encountered & resolved

- **Postgres 18 refused to start** on the first attempt with the old volume mount (`/var/lib/postgresql/data`). The image's startup script detected an "unused mount" at the legacy path. Resolved by moving the mount to `/var/lib/postgresql` and recreating the volume.
- **`docker compose restart` did not pick up the realm JSON volume change**: restart preserves the container config, only stop/start of the process. Fixed by using `up -d --force-recreate keycloak`.
- **Keycloak admin console login failed with `user_not_found`** when trying `testuser`. Cause: admin console uses the `master` realm, not `pecunia`. The intended login flow for `testuser` will go through Spring Security against the `pecunia` realm.
- **Keycloak `/realms/pecunia/account` returned "Something went wrong"** after a successful login. This is a known quirk of the built-in account console against minimal realm imports; it does not affect the OAuth2 authorization-code flow the application uses. Deferred.
- **The assistant initially suggested `spring.session.store-type: redis`** without verifying ‚Äî user correctly identified it as removed since Spring Boot 3.0, web-search confirmed.
