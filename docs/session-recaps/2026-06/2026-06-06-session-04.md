## Key decisions made

- **`SecurityConfig` lives in `com.pecunia.shared.security`** (option B
  among four package candidates). Reason: aligns with the modular
  monolith intent — `shared` is the cross-cutting kernel that future
  business modules (`accounts`, `transactions`, …) will reference, and
  security belongs there. Anticipates companions
  (`UserContextHolder`, MDC filter, ownership exception mappers).
- **Plain `CsrfTokenRequestAttributeHandler` over the default Xor**.
  Reason: BREACH attack surface is closed by architecture (token lives
  in a cookie, never rendered in a compressible HTML body) — Xor
  defends against an attack that does not apply to BFF/SPA. Plain
  handler integrates natively with Angular's `HttpXsrfTokenExtractor`.
- **`csrf(CsrfConfigurer::spa)` replacing the manual three-part
  setup**. Reason: idiomatic Spring Security 6.5+/7 shortcut, single
  declarative line, forward-compatible with framework evolution.
  Captured in ADR-0022.
- **CORS deferred to Block 1 step 7** (when Angular arrives). Reason:
  YAGNI — no concrete frontend yet, allowed origin and methods are
  best decided with real `ng serve` config in hand.
- **Spotless scope: Java + `pom.xml` (sortPom conservative)**. Reason:
  Java formatting is the primary value; `sortPom` normalizes
  whitespace/indentation drift without breaking the human-organized
  comment-grouped dependency blocks. YAML deferred.
- **Spotless `check` on `verify`, `apply` manual**. Reason: fail-fast
  in CI; auto-apply during builds would surprise developers and
  trigger DevTools restarts.
- **No root `pom.xml`**. Reason: would falsely signal that the entire
  polyglot monorepo is a Maven project, creating an asymmetry with
  `apps/frontend` (npm). Tradeoff accepted: `mvn -pl` cannot be used,
  documentation and habits switch to `-f`.

## Concepts explained / learned

- **Spring Security `DeferredCsrfToken`**: in Security 6+, the token
  is generated lazily — only when `CsrfToken#getToken()` is actually
  invoked. On simple GETs the token never materializes and the
  `XSRF-TOKEN` cookie is never written. To make the cookie available
  to a SPA, either a custom `CsrfCookieFilter` calls `getToken()` on
  every request, or `csrf.spa()` does it under the hood.
- **CSRF token attribute keys**: `CsrfTokenRequestAttributeHandler`
  stores the token under two request attributes — the FQN
  `CsrfToken.class.getName()` (canonical, stable) and `"_csrf"`
  (configurable, defaults to the parameter name). The canonical FQN
  is the defensive choice for custom filters.
- **BREACH attack mechanics**: compression-side-channel that extracts
  secrets byte-by-byte from HTTPS response bodies. Requires the
  secret to be rendered in a compressible body alongside attacker-
  controlled reflected content. Does **not** apply when the token
  lives in a cookie (header, not body, and headers aren't gzipped in
  HTTP/1.1).
- **Spring Boot 4 actuator package reorganization**: imports
  confirmed Session 03's finding — `HealthEndpoint` is now under
  `org.springframework.boot.health.actuate.endpoint`, `EndpointRequest`
  under `org.springframework.boot.security.autoconfigure.actuate.web.servlet`.
  Per-feature modules replace the monolithic `spring-boot-actuator-autoconfigure`.
- **Palantir Java Format uses internal JDK APIs** (`com.sun.tools.javac.*`)
  that require `--add-exports` JVM flags at runtime. Spotless 2.x
  handles flag injection automatically, but Palantir releases must
  track new JDK versions when those internals change signatures
  (as happened between Java 24 and Java 25).
- **Maven `-pl` and the reactor**: `-pl <module>` only works inside a
  multi-module reactor — i.e., when a parent pom at the run location
  declares `<modules>`. Without one, `-pl` errors out before the goal
  even starts.

## Issues encountered & resolved

- **`SecurityConfig` first draft used `/actuator/**` path-based
  matcher** which conflicted with ADR-0021's allowlist-by-default
  posture and would silently grant `permitAll` to any future endpoint
  added to the YAML exposure list. Fixed by switching to the typed
  matcher `EndpointRequest.to(HealthEndpoint.class, InfoEndpoint.class)`.
- **Post-login WhiteLabel page** on `/` was initially worrying but
  diagnosed as `NoResourceFoundException` raised by Spring MVC after
  the security chain successfully authenticated the user. Not a
  security bug; just the absence of a controller/static at `/`.
- **`XSRF-TOKEN` cookie missing** in Chrome DevTools after login.
  Root cause: Spring Security 6's `DeferredCsrfToken` design — the
  cookie is only written when something materializes the token, which
  does not happen on bare GETs. Initially resolved with a custom
  `CsrfCookieFilter`; later superseded by the `csrf.spa()` shortcut.
- **`mvn -pl apps/api spotless:apply` failure** ("Could not find the
  selected project in the reactor"). Root cause: no root pom.
  Resolved by switching to `mvn -f apps/api/pom.xml …` and fixing
  all docs that referenced the broken form.
- **`spotless:apply` second failure** with `NoSuchMethodError` on
  `com.sun.tools.javac.util.Log$DeferredDiagnosticHandler.getDiagnostics()`
  — Palantir Java Format 2.55.0 doesn't support Java 25's javac
  internals. Fixed by bumping to 2.91.0 (latest Maven Central).
