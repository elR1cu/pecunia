## Key decisions made

- **Use Spring Boot 4 native structured logging, not
  `logstash-logback-encoder`.** Same JSON wire format, zero external
  dependency, configured in YAML, evolves with the framework. The
  third-party library remains a fallback if SB4 native hits a bug.
- **Use Micrometer Tracing via `spring-boot-starter-opentelemetry`
  for correlation IDs, not a custom `RequestIdFilter`.** The bridge
  populates `traceId`/`spanId` in MDC automatically, propagates W3C
  `traceparent`, handles async / reactive context correctly, and
  provides a clean path to span export in Block 9. The custom filter
  pattern is preserved in ADR-0018's "Alternatives Considered" with
  the full reference code as a learning artifact.
- **Force tracing sampling to 1.0.** Micrometer's 10% default would
  leave most requests without a traceId at Pecunia's single-user
  scale — unacceptable.
- **Disable OTLP exporters at Block 1.** Spans / logs / metrics are
  generated locally (MDC reaches consumers) but nothing is shipped
  over the wire. Block 9 will flip the tracing flag and configure an
  endpoint.
- **Opt in to `%kvp` in the dev console pattern.** Spring Boot
  rejected including it by default for security reasons; we opt in
  deliberately and rely on the sanitization discipline (no PII as
  KVPs) plus the future Sub-session C customizer.

## Concepts explained / learned

- **`logging.structured.format.console=logstash` (SB4)** : native
  Spring Boot 4 path that produces the same JSON wire format as
  `logstash-logback-encoder` without any third-party dependency. MDC
  entries and SLF4J fluent `addKeyValue` pairs are promoted to
  top-level JSON fields automatically.
- **Asymmetric OTLP property namespace** : SB4 routes traces and logs
  through the OpenTelemetry SDK (configured under
  `management.opentelemetry.*.export.otlp.*`) while keeping metrics on
  Micrometer's `OtlpMeterRegistry` (configured under
  `management.otlp.metrics.export.*`). Both end up as OTLP on the wire
  but the in-process pipeline differs.
- **OTLP = OpenTelemetry Protocol** : vendor-neutral wire protocol
  (protobuf binary, gRPC :4317 or HTTP :4318) that replaces
  proprietary ingestion formats (Jaeger Thrift, Zipkin B3, Datadog
  API…). Same data, one protocol — choose your backend.
- **SLF4J 2.x fluent API** :
  `log.atInfo().addKeyValue("k", v).log("msg")` separates the
  structured key-value payload from the message text. JSON output
  promotes `k` to a top-level field; text output needs `%kvp` in the
  pattern (not on by default).
- **Spring Boot 4 autoconfig split** : confirmed once again — bare
  `micrometer-tracing-bridge-otel` brings the runtime but not the
  autoconfig (which lives in a separate
  `spring-boot-micrometer-tracing-opentelemetry` module). The
  `spring-boot-starter-opentelemetry` starter is the one-line SB4
  way to get everything wired.
- **`ServerHttpObservationFilter` is where the span starts** for HTTP
  requests in SB. Logs emitted before this filter runs (startup
  banner, Tomcat connector, scheduler beans) legitimately have no
  `traceId` — the `%X{traceId:-}` empty-default pattern renders them
  as empty rather than crashing.

## Issues encountered & resolved

- **`logstash-logback-encoder` proposed before verifying SB4
  alternatives.** Original Sub-session A proposal jumped to the
  third-party library, missing that Spring Boot 3.4+ ships native
  structured logging. Self-corrected after research; ADR-0018
  revised.
- **Custom `RequestIdFilter` proposed before verifying SB4
  alternatives.** Same pattern, second time in the same session.
  Self-corrected after research; ADR-0018 revised again. Lesson
  internalized: search SB4 docs *before* proposing servlet-level
  patterns.
- **Bare `micrometer-tracing-bridge-otel` failed silently.** Added
  it as the "minimal" dependency; restarted the app; `traceId` stayed
  empty. Diagnosed as the SB4 autoconfig split: the bridge needs the
  matching `spring-boot-micrometer-tracing-opentelemetry` autoconfig
  module, which only `spring-boot-starter-opentelemetry` bundles.
  Swapped the dep.
- **`management.otlp.tracing.export.enabled` is not a real property
  in SB4.** Initial config used this prefix on the assumption it
  mirrored metrics; Spring Boot silently ignored it. Verified against
  the SB4 application-properties appendix: tracing moved to
  `management.opentelemetry.tracing.export.otlp.*` while metrics
  stayed at `management.otlp.metrics.export.*`. Fixed.
- **`traceId` appearing empty on initial probes.** User reported empty
  traces for `/me`; turned out they were checking logs emitted before
  the controller (startup / framework code outside any HTTP request).
  Adding a log inside `MeController` showed `traceId`/`spanId`
  correctly populated. Behavior is correct.
- **`addKeyValue(...)` not visible in dev text pattern.** Expected:
  Spring Boot rejected adding `%kvp` to the default pattern for
  security reasons (PR #47162). Opted in by overriding
  `logging.pattern.console` in `application-dev.yml`.
