## Key decisions made
- **Auth protection = A + C** (guard + global 401 interceptor) rather than guard
  alone. The interceptor also covers mid-session expiry on any future API call,
  not just route entry; it's ~10 lines and teaches functional interceptors.
- **Interceptor owns the redirect, guard only allows/blocks.** The guard's own
  `getCurrentUser()` 401 also passes through the interceptor, so duplicating the
  redirect in the guard would be non-DRY.
- **Post-login landing via backend `defaultSuccessUrl("/dashboard", true)`**
  (Option 2) rather than a frontend check on the landing (Option 1). Option 1
  collides with the global interceptor: the landing's `getCurrentUser()` would
  401 for anonymous visitors and redirect them away from the public page.
- **Narrow Keycloak redirect URIs**: only the exact callback, no `:4200/*`
  wildcard (security; the realm is versioned and will be public).
- **Frontend test policy applied**: test the branching glue (guard, interceptor),
  drop "should create" stubs on purely structural components. Candidate for a
  small ADR later.

## Concepts explained / learned
- **Functional `HttpInterceptorFn`**: `(req, next)`; react to responses in
  `next(req).pipe(catchError(...))`; `inject()` works (injection context);
  rethrow with `throwError(() => error)` to keep the caller's error path alive;
  onion ordering in `withInterceptors([...])`.
- **`CanActivateFn`**: returns `boolean | UrlTree | Observable/Promise` thereof;
  the router awaits the observable's first value; `inject()` available; an
  HttpClient observable completes after one emission so the router doesn't hang.
- **`toSignal()`**: bridges Observable ‚Üí Signal, auto-subscribes/unsubscribes on
  the component lifecycle (no manual teardown), initial value `undefined` until
  the HTTP resolves (hence `@if`/optional chaining in the template).
- **Two testing techniques**: `HttpTestingController` + `provideHttpClientTesting`
  (`expectOne`/`flush`) to drive the interceptor through a real-but-mocked HTTP
  pipeline; DI fake (`{ provide: IdentityService, useValue: {...} }`) to isolate
  the guard's decision logic. `runInInjectionContext` to execute a functional
  guard in tests.

## Issues encountered & resolved
- **Keycloak ignored the edited realm JSON.** The dev Keycloak persists the
  realm in Postgres and `--import-realm` does not overwrite an existing realm;
  editing the JSON + restart had no effect. Resolved with
  `docker compose ... down -v && up -d` to force a fresh import. Saved as memory
  `project-keycloak-realm-reimport`.
- **`dashboard.spec.ts` (generated stub) broke the test build**: creating the
  component injects `IdentityService` ‚Üí `HttpClient`, absent from the stub's
  `TestBed`. Deleted (Option 1).
- **`app.spec.ts` "should render title" failed** after `app.html` was reduced to
  `<router-outlet />` (no more `<h1>`). Deleted as an obsolete shell stub; also
  removed the now-unused `signal` import from `app.ts`.
- **Process slip**: first version of the generated tests had French comments;
  corrected to English per CLAUDE.md (code comments in English), and fixed `/me`
  ‚Üí `/api/me` in the guard test descriptions.
