## Key decisions made

- **V1__init.sql is a portable baseline** (`DO $$ … $$` + `current_database()`)
  rather than `ALTER DATABASE pecunia …` hardcoded. Reason: identical
  migration must run against the dev database and against the
  Testcontainers-created database (typically `test`) when integration
  tests land in Block 2.
- **`spring-boot-docker-compose` over Testcontainers dev mode or IDE
  Before-Launch hooks** for managing the dev stack lifecycle. Reasons:
  preserves the Keycloak realm import (~5 s) and Postgres state across
  app restarts; teaches the *service connections* abstraction reused by
  Testcontainers in tests; idiomatic Spring Boot 3.1+/4. Captured in
  ADR-0020.
- **`lifecycle-management: start-only`** rather than the default
  `start-and-stop`. Reason: Ctrl+C of Spring should not tear down the
  stack; the developer stops it explicitly when wanted (a one-line
  `docker compose … down`).
- **Working directory = repo root** for every run method (CLI and
  IntelliJ Run Configuration). Reason: both relative paths
  (`apps/api/.env` for `spring.config.import` and
  `deploy/docker-compose/…` for `spring.docker.compose.file`) must
  resolve from the same `user.dir`.
- **Spring Boot DevTools added to the pom**. Reason: developer
  productivity, classpath-change restart is much faster than full JVM
  restart. Trade-off accepted: IntelliJ requires a one-off two-setting
  configuration to make it fire.
- **Actuator two-layer policy** (HTTP exposure allowlist at the Actuator
  level + tiered access policy at the Spring Security level). Reason:
  defense in depth. If the security layer is misconfigured, an
  unexposed endpoint cannot leak. Captured in ADR-0021.
- **`show-details: when-authorized`** for `/actuator/health`. Reason:
  tightest setting that still gives a logged-in operator full component
  visibility without leaking the dependency inventory to anonymous
  callers.

## Concepts explained / learned

- **Spring Boot service connections** are a single abstraction that both
  `spring-boot-docker-compose` and Testcontainers (with `@ServiceConnection`)
  produce. Knowing one transfers directly to the other.
- **`ALTER DATABASE … SET` semantics**: the change applies to *new*
  sessions only. In our flow this is fine because Hikari opens
  connections lazily *after* Flyway runs.
- **Flyway naming convention**: `V<version>__<description>.sql` with
  *two* underscores. A single underscore = `ValidateException` at boot.
- **Spring Boot DevTools needs a classpath change to fire**, and
  IntelliJ does not recompile by default while an app is running. Two
  one-off settings (`Build project automatically` + `Allow auto-make to
  start even if developed application is currently running`) unlock the
  loop.
- **Why `/actuator/health` requires login by default**: with
  `spring-boot-starter-oauth2-client` on the classpath and no custom
  `SecurityFilterChain`, Spring auto-configures `anyRequest().authenticated()`
  with `oauth2Login()` as the entry point. The actuator exposure and
  `show-details` properties control *whether the endpoint exists* and
  *what it returns*, not *who can call it*.
- **`current_database()` + `format('%I', …)`**: the canonical PostgreSQL
  pattern to build dynamic DDL that quotes the identifier safely.
- **Keycloak has no built-in `ConnectionDetails` factory in Spring Boot
  4**: the container is started by `spring-boot-docker-compose` but the
  OIDC configuration (`issuer-uri`) stays driven by `application.yml`.
  For Postgres and Redis, the service connection silently overrides the
  YAML — important to know when debugging config drift.
- **Spring Boot 4 auto-configurations live in dedicated modules**, not in
  the monolithic `spring-boot-autoconfigure` jar anymore. The classpath
  shows the new shape: `spring-boot-flyway`, `spring-boot-data-jpa`,
  `spring-boot-jdbc`, `spring-boot-security`, `spring-boot-health`,
  `spring-boot-tomcat`, etc. Consequence: adding only the third-party lib
  (e.g. `flyway-core`) gives the runtime code but no autoconfig — the
  feature silently does nothing at boot, with no error log. Rule of
  thumb: prefer `spring-boot-starter-<feature>` to the raw dep whenever
  Spring offers one; keep direct deps only for things the starter
  doesn't cover (driver-specific modules like `flyway-database-postgresql`).
