## Key decisions made

- **Anticipated blacklist over compact**: include PII / finance field
  names (`iban`, `account`, `card`, `cvv`, `ssn`, `email`, `phone`,
  `address`, `amount`) from C-min rather than wait for Block 2/3.
  Rationale: defence in depth; the cost is near-zero (`Set.contains`
  on names that don't yet flow through KVPs), the upside is no
  regression window when the data starts flowing.
- **Custom `MaskedKvpConverter` over native Logback `%maskedKvp`**:
  the Logback-native converter exists since 1.5.7 but takes its
  blacklist as **pattern options in XML** (drift risk vs. the Java
  blacklist) and matches **case-sensitively** (asymmetry with the
  JSON path's `Locale.ROOT` matching). The ~30 lines of custom Java
  delegating to `SensitiveFieldNames` preserve a single source of
  truth.
- **Exact match, case-insensitive on field names** (not contains).
  Trade-off: `accountId` / `accountStatus` are deliberately
  **not** masked when `account` is in the blacklist. Documented in
  `SensitiveFieldNames`' Javadoc.
- **Inline correlation triplet in the dev console pattern** rather
  than indirection through `${LOG_CORRELATION_PATTERN:-}`. Tried both
  the System-property path (SB4 default with custom XML) and
  `<springProperty scope="context">` (recommended workaround) —
  neither propagated the value to the encoder. Inlining
  `[${spring.application.name:},%X{traceId:-},%X{spanId:-}]`
  bypasses the moving parts.
- **`logback-spring.xml` carries the full Spring Boot include
  ceremony**: `defaults.xml` + conditional `console-appender.xml` /
  `structured-console-appender.xml` by profile + explicit root logger.
  When SB4 detects a custom XML, it stops auto-wiring the CONSOLE
  appender — declared from the file is mandatory. Per
  [SB Issue #38687](https://github.com/spring-projects/spring-boot/issues/38687),
  no shorter form exists for registering a custom Logback conversion
  rule.

## Concepts explained / learned

- **`StructuredLoggingJsonMembersCustomizer` SPI**:
  `void customize(JsonWriter.Members<T> members)`. `Members` exposes
  `applyingValueProcessor(ValueProcessor<?>)`, where `ValueProcessor`
  is `(MemberPath path, T value) -> T`. `path.name()` gives the last
  segment name. The Logstash formatter flattens MDC entries and KVPs
  as top-level members, so a fluent
  `log.atInfo().addKeyValue("password", "x").log(...)` reaches the
  processor with `path.name() == "password"`. Top-level structural
  fields (`@timestamp`, `message`, `logger_name`, …) traverse the
  processor too but never match the blacklist by construction.
- **Logback `ClassicConverter`**: abstract single-method class
  (`String convert(ILoggingEvent event)`). Stateless by contract
  (instantiated once, called concurrently). Registered via
  `<conversionRule conversionWord="X" class="...converter.Class"/>`
  in `logback-spring.xml`. The attribute name moved from
  `converterClass` (deprecated, still accepted) to `class` in modern
  Logback.
- **Native Logback `%maskedKvp`** (`MaskedKeyValuePairConverter`
  since 1.5.7): masks values whose **keys** match an option list
  passed in-pattern. Case-sensitive `List.contains` — no
  case-insensitivity, no shared-with-Java blacklist. Not a fit when
  the JSON side already has a Java-driven, case-insensitive
  blacklist.
- **Logback 1.5.22+ variable auto-masking**: hides values of
  `${...}` substitutions in Logback config whose **variable name**
  contains `password`, `secret`, or `confidential`. Targets config-
  time secret leakage, **not** runtime KVP / MDC masking. Orthogonal
  to our customizer.
- **SB4 `LogbackLoggingSystem` asymmetry**: `loadDefaults()` (no
  custom XML) pushes `LOG_*` properties onto the `LoggerContext`
  directly (`loggerContext::putProperty`). `loadConfiguration()`
  (custom XML) pushes them via `System.setProperty` only. The
  `${LOG_CORRELATION_PATTERN:-}` substitution in our dev pattern then
  rebut empty depending on Logback's parse phase. Cause noted in
  the new memory; visible only in the dev text pattern, JSON output
  unaffected.
- **`<springProperty>`**: SB Joran extension that reads from the
  Spring `Environment` and sets a Logback property at the requested
  scope. Implementation in `SpringPropertyModelHandler` confirms the
  value is taken from `environment.getProperty(source, defaultValue)`
  with Spring placeholder substitution. **Did not work here** as a
  workaround for the indirection — likely the property is set after
  the encoder pattern is compiled. The inline-pattern fix sidesteps
  the question.
- **Verbosity of `logback-spring.xml` is intrinsic**: with no SB
  property-based way to register a conversion rule (issue #38687
  open, no design yet), the file must include `defaults.xml`, pick
  a CONSOLE appender, and declare the root logger. Each line earns
  its place.

## Issues encountered & resolved

- **First `logback-spring.xml` only declared `<conversionRule>` and
  nothing else.** Spring Boot, upon detecting the file, stopped
  auto-wiring its CONSOLE appender. The app started, emitted the
  banner attempt, then exited silently with only
  `WARN in Logger[com.pecunia.PecuniaApplication] - No appenders
  present in context [default]`. Resolved by including
  `defaults.xml`, picking the appropriate console appender by
  profile, and declaring an explicit `<root>` logger.
- **Bash `mvn spring-boot:run` testing was misleading**: launching
  the app from a non-interactive Bash session (background or not)
  returned BUILD SUCCESS in ~3 seconds with no banner — the JVM
  appeared to exit immediately and silently. Could not reproduce
  the real startup environment. Resolved by switching to user-driven
  IntelliJ runs for validation.
- **Docker-compose path resolution failed at app startup** once the
  `logback-spring.xml` was good enough to surface errors:
  `'files' content [deploy/docker-compose/docker-compose.dev.yml]
  must exist`. The error pre-existed our logging work; it had been
  invisible because Logback had no appender to ship the error
  through. The `spring-boot-maven-plugin` already had
  `<workingDirectory>${project.basedir}/../..</workingDirectory>` —
  the diagnostic of "missing workingDirectory" was wrong on my part
  (I had only grep'd 5 lines of context in the pom and missed it).
  Actually fixed: user had added the `workingDirectory` independently
  in parallel before I re-checked.
- **`LOG_CORRELATION_PATTERN` regression** (introduced by the
  `logback-spring.xml`): the dev pattern's
  `${LOG_CORRELATION_PATTERN:-}` resolved empty. First fix attempt
  (`<springProperty scope="context">`): no effect, same regression.
  Second fix (chosen): inline the triplet directly in the
  `logging.pattern.console` value. Root cause confirmed in
  `LogbackLoggingSystem` source (custom-XML path uses
  `System.setProperty`, no-XML path uses
  `LoggerContext.putProperty`).
- **End-to-end masking validation succeeded on all six scenarios**
  across both dev text and prod JSON: non-sensitive KVP passes
  through; `password` / `iban` masked; case-insensitive
  (`PASSWORD` masked); exact-match preserves `accountId`; structural
  JSON fields untouched; `traceId` / `spanId` correctly populated
  from the Micrometer Tracing MDC.
