## Key decisions made

- **A1 — POJOs for generated DTOs.** Records not natively supported by
  the spring generator; the only workaround is the preview-stage Chrimle
  Mustache templates (external maintainer, fragile across plugin
  upgrades). Generated POJOs stay confined to the `generated`/`dto`
  packages, are never edited, and MapStruct bridges them to records in
  the domain layer.
- **B — Static spec served, no runtime introspection.** `contracts/openapi.yaml`
  is the single source of truth; Springdoc's runtime introspection
  (`/v3/api-docs`) is disabled to prevent silent drift. Corollary:
  `documentationProvider: none` in the codegen plugin makes generated
  interfaces free of Springdoc annotations that would be dead weight.
- **C — Module/tag package layout, multi-`<execution>`.** Each tag/module
  gets its own openapi-generator execution. For `Identity`, one execution
  scoped via `globalProperties.apis=Identity`. As modules grow, models
  shared across tags will need explicit allocation; for now, only
  `CurrentUser` exists and lives in the Identity execution.
- **D-α — Swagger UI dev-only, anonymous.** Swagger UI is a developer
  productivity tool, not a feature shipped to users. Disabled in prod
  (`springdoc.swagger-ui.enabled: false` by default, overridden in
  `application-dev.yml`). In dev, anonymous access on the Swagger paths
  to avoid login friction during iteration.
- **E1 — Copy YAML to classpath via maven-resources-plugin.** Avoids
  introducing a small `WebMvcConfigurer` (which would be applicative
  code) and means the YAML travels with the JAR. Slight duplication at
  runtime (repo + `target/classes/static/`), but mechanically standard.
- **F1 — Dev profile + `@Profile("dev") SwaggerSecurityConfig`.** A
  second `SecurityFilterChain` bean, scoped via `securityMatcher` to the
  Swagger paths, with `@Order(0)` priority over the main chain. The main
  `SecurityConfig` chain is not touched. Total cost: ~15 lines plus
  adopting the Spring profile convention (which would be needed soon
  anyway for prod overrides).
- **G — Skip realm-level emailRequired for MVP.** No `emailRequired`
  field exists at the Keycloak realm root; the modern path is the
  Declarative User Profile (~20-30 lines of poorly documented nested
  JSON). With `registrationAllowed: false` and the testuser already
  carrying an email, the operational guarantee is satisfied. Contract
  enforcement lives in the OpenAPI spec (`email` required) and will fail
  loud in `MeController` if Keycloak ever omits the claim.
- **H — `CurrentUser.id` = Keycloak `sub` for now.** Block 1 is
  intentionally a thin "secured skeleton"; introducing a `User` entity
  with provisioning belongs in Block 2 (Domain Model). The refactor when
  the entity arrives is small: only `MeController` and the source of the
  `id` field change; the DTO shape stays UUID-typed.
- **I — Cookie security scheme (`bffSession`).** `type: apiKey`, `in:
  cookie`, `name: SESSION`. Documents reality (BFF + session cookie) over
  what's tempting (OIDC client-side, which is not how a BFF works). Cost
  is 4 lines of YAML for genuine spec honesty.
- **Schema name `CurrentUser`** over `Me` / `UserProfile` / `MeResponse`.
  Singular, no `Response` suffix (good OpenAPI hygiene), semantically
  clear, doesn't collide with a future `User` domain entity.
- **`MeController` package: `com.pecunia.identity.api`** (Option 2).
  Co-located with the generated interface it implements; the `api/`
  sub-package becomes the HTTP-boundary umbrella, with `generated/` and
  `dto/` isolating auto-generated code.

## Concepts explained / learned

- **`useSpringBoot4` + `useJackson3` codegen options**: the dedicated
  flags introduced in openapi-generator 7.x for Spring Boot 4 support.
  Setting `useSpringBoot4=true` auto-enables `useJakartaEe`; setting
  `useJackson3=true` requires `useSpringBoot4`.
- **Record support gap in the spring generator**: issue #10490 has been
  open since 2021. The Java generator (client) similarly lacks the
  option. The only path to generated records is the third-party
  `chrimle/openapi-to-java-records-mustache-templates`, marked
  experimental for the spring generator.
- **MapStruct + records**: MapStruct ≥ 1.5 detects records and calls the
  canonical constructor directly with component-name matching. No
  config required.
- **`documentationProvider: none` corollary of static spec serving**: in
  schema-first + static spec mode, runtime Springdoc annotations on
  controllers/interfaces become dead weight. Suppressing them keeps the
  generated interfaces lean.
- **`skipDefaultInterface: true`**: the spring generator's default
  behavior is to emit method bodies on the interface that return an
  example from the spec via `ApiUtil.setExampleResponse(...)`. Useful
  for quick mocking; useless when implementing the methods. Suppressing
  default bodies removes the `ApiUtil` dependency that we also suppress
  via `generateSupportingFiles=false`.
- **`DelegatingAuthenticationEntryPoint`**: Spring Security's built-in
  mechanism to discriminate the "unauthenticated request" response by
  request shape. The canonical BFF pattern matches the `Accept` header:
  `application/json` → `HttpStatusEntryPoint(401)`, everything else
  falls back to `LoginUrlAuthenticationEntryPoint("/oauth2/authorization/pecunia")`
  (i.e., 302 redirect). Returning 302 to a fetch call breaks the SPA
  (the redirect lands on Keycloak HTML the fetch can't parse, or
  triggers a CORS error); returning 401 lets the SPA detect "no
  session" cleanly and trigger its own redirect.
- **Generated interface signature constraint**: `IdentityApi.getCurrentUser()`
  takes no parameters. Adding `@AuthenticationPrincipal OidcUser user`
  on the implementation override is not a real Java override (different
  signature); the only path with a fixed interface is
  `SecurityContextHolder.getContext().getAuthentication().getPrincipal()`.
- **`DeferredCsrfToken` and `csrf.spa()`**: confirmed from Session 04 —
  the SPA shortcut bundles cookie repository + plain handler + the
  filter that materializes the token on every request.
- **Springdoc 3.0.3 / Spring Boot 4 compat**: confirmed via springdoc.org/v4/.
  The Jackson 3 compat issue (#3200) was a 3.0.1 problem; 3.0.3 is the
  current release.
- **`emailVerified` claim**: separate from the `email` claim itself.
  The current `MeController` plan reads `getEmail()` only; whether to
  also gate on `getEmailVerified()` is an open question.

## Issues encountered & resolved

- **`IdentityApi.java cannot find symbol: variable ApiUtil`**.
  Cause: `generateSupportingFiles: false` suppresses the `ApiUtil`
  helper, but the generator still emitted default method bodies that
  reference it. Resolved by adding `skipDefaultInterface: true` to
  `configOptions` (which is what we want anyway: we implement the
  methods, no mock bodies needed).
- **Incorrect early advice on `@AuthenticationPrincipal`**. Initially
  suggested it could be added as a method parameter on the
  `IdentityApi.getCurrentUser()` override. Wrong: Java's override
  contract forbids adding parameters. Corrected to the
  `SecurityContextHolder` idiom.
- **Outdated `useSpringBoot3` default**. Initially defaulted to
  `useSpringBoot3: true` in the configuration; user pointed out
  `useSpringBoot4` exists. Research confirmed and the config was
  corrected. Memory `feedback-spring-boot-4-docs` from Session 04
  flagged this exact failure mode; should have applied it earlier in
  this session.
- **`emailRequired: true` doesn't exist at realm level in Keycloak**.
  The user explicitly asked for the modification; investigation
  revealed the modern path is the Declarative User Profile (verbose,
  poorly documented, version-sensitive). Surfaced the situation and
  the user accepted the YAGNI skip for MVP.
- **OpenAPI 3.1 beta warning**: codegen logs `[WARNING] OpenAPI 3.1
  support is still in beta`. Non-fatal; generation succeeded. Could be
  silenced by downgrading the spec to `openapi: 3.0.3`; left as-is for
  now.
