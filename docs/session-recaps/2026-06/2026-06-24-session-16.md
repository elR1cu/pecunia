## Key decisions made
- **i18n technology already fixed by ADR-0025** (ngx-translate v18); no new
  tech decision, just implementation.
- **No initial `lang` in `provideTranslateService`** (only `fallbackLang: en`)
  so `LanguageService.init()` picks the persisted language once, avoiding a
  wasted default-file fetch.
- **`/api/me` dedup via a signal state service** (Option A) over a resolver or
  a bare `shareReplay`: identity is cross-cutting app state, signals-first;
  and BFF full-page login/logout reloads reset the cache for free (no manual
  invalidation needed for the MVP).
- **Frontend SonarCloud as a separate project (monorepo)**, the SonarCloud-
  documented best practice, over a single unified scan — keeps the working
  backend Maven integration untouched.
- **Repo made public** to unlock the OSS plan (free unlimited branch/PR
  analysis); the free plan covers only `main` for private repos.
- **CSRF S4502 resolved as a UI False Positive** for now (user's call), with
  the durable in-pom `sonar.issue.ignore.multicriteria` suppression noted as
  the alternative that survives project recreation.
- **Language switcher placed under `components/`** (plural bucket, matching
  `guards/`/`services/`), not a flat folder or `shared/`.

## Concepts explained / learned
- **ngx-translate v18 is signals-native**: `currentLang` is a `Signal`, the
  `TranslatePipe` is signal-backed (updates on `use()` without reload).
- **`@angular/build:unit-test` coverage options are `coverage` /
  `coverageReporters`** (not the `codeCoverage*` names in newer docs);
  requires `@vitest/coverage-v8`; LCOV lands at `coverage/pecunia-frontend/`.
- **SonarCloud project keys are case-sensitive**: the CI used `elR1cu_…`
  (capital R) while the org slug is `elr1cu` — manual recreation with the
  lowercase slug created *parallel* projects. This was the root cause of the
  "duplicate / not displayed" confusion.
- **A SonarCloud analysis is a full snapshot that replaces the previous** →
  one project per build tool (separate scans) or a single unified scan; two
  partial scans to one project clobber each other.
- **A UI "False Positive" resolution lives in the project** and is lost on
  project deletion/recreation; an in-config suppression is version-controlled
  and durable.
- **SonarCloud free plan = main-branch-only for private repos**; PR/branch
  analysis is free only on public (OSS) repos or paid plans.
- **`sonar.projectName`** sets the display name without changing the
  (immutable) project key.

## Issues encountered & resolved
- **Backend Quality Gate red** on the recreated project: `new_security_rating`
  E from the CSRF `S4502` on `SwaggerSecurityConfig` (the FP did not migrate
  from the old project). Resolved by re-marking it False Positive once the
  project was public.
- **"Project doesn't exist" / "main branch only"**: the projects were still
  private (GitHub repo public ≠ SonarCloud project public) and the case
  mismatch had created duplicates. Resolved by making both projects public,
  deleting the lowercase duplicates, and keeping the capital-R keys + adding
  `projectName`.
- **`codeCoverage` schema error**: wrong option name for the installed builder
  version; corrected to `coverage` / `coverageReporters` by reading the
  builder schema.
- **`scan-action@v5` was stale**: bumped to `@v8` (latest; non-Docker, GPG-
  validated).
- **Default-private auto-provisioning**: the scanner auto-creates the project
  as private; reruns can't recreate monorepo projects (manual creation
  required), so reruns alone never unblocked anything.
