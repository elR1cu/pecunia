## Key decisions made

- **`npx @angular/cli@22` (no global install)**. Rationale: no
  pollution of the global npm prefix, version explicit on each
  scaffolding invocation, no drift if a second Angular project
  appears later. Cost: `ng` is invoked through `npm start` /
  `npm run` / `npx`, never bare.
- **Zoneless change detection**. Rationale: zone.js is the
  historical compatibility shim; with Signals-first state
  management already committed (per `docs/architecture.md`), the
  zoneless mode is the conceptually consistent option. Angular 22
  has zoneless as the *default*, so no provider call is needed in
  `app.config.ts` ŌĆö only the absence of `zone.js` from the deps.
- **SCSS over plain CSS**. Rationale: Angular Material's theming
  layer is SCSS-native (mixins, design-tokens). Going SCSS now
  avoids reconfiguring later when Material lands.
- **SPA only, no SSR**. Rationale: BFF sessions are authenticated
  and personal; there is nothing to pre-render. SSR would add a
  Node runtime to the prod deployment without offsetting benefit.
- **Node 24.16.0 LTS (Krypton)** over 22.22.3. Rationale: longest
  active LTS support window (until ~April 2028), no breaking
  change for our tooling. The patch upgrade route within Node 22
  was equally valid but offered a shorter horizon.
- **`.nvmrc` scoped to `apps/frontend/`** rather than the repo
  root. Rationale: the repo is polyglot; the backend is Java
  and has no Node dependency. A root `.nvmrc` would
  mislead.
- **Versioning `apps/frontend/.vscode/`** (extensions, launch,
  tasks, mcp). Rationale: those files document recommended
  extensions (`angular.ng-template`), debug configurations
  (Chrome attach to `ng serve`), and the new MCP server
  declaration (`@angular/cli mcp`) ŌĆö all benefit other
  contributors and have no secret content.
- **Conventional Commit shape**: two commits, `chore:` for the
  root `.gitignore` tweak (not a build-system change), and
  `feat(frontend):` for the application scaffold. The Angular
  CLI scaffold was treated as the introduction of a new project
  feature, consistent with the prior `feat(api): scaffold OIDC
  login ŌĆ”` choice on the backend side.

## Concepts explained / learned

- **Angular CLI 22 brings several defaults that matter for
  Pecunia**:
  - **Vitest 4 as the test runner**, replacing
    Karma/Jasmine. Karma reached end-of-life on the Angular
    side, and Vitest aligns with the broader JS testing
    ecosystem (no browser launcher needed by default, jsdom
    is the DOM, watcher and reporter ergonomics are
    superior). The generated `package.json` ships
    `vitest@4` + `jsdom@28` and the builder target uses
    `@angular/build:unit-test`.
  - **Naming convention without the `.component` suffix**.
    Files are `app.ts`, `app.html`, `app.scss`,
    `app.spec.ts`; the class is `App`, not `AppComponent`.
    Effective since Angular 20.
  - **`public/` instead of `src/assets/`** for static files
    (favicon, images, etc.). The build pipeline maps the
    folder via the `assets` array in `angular.json`.
- **Zoneless is the v21+ default**. The Angular doc explicitly
  states: *"Zoneless is the default in Angular v21+ so you do
  not need to do anything to enable it. You should verify that
  `provideZoneChangeDetection` is not used anywhere to override
  the default configuration."* Removing `zone.js` from
  `package.json` is the only action needed.
- **TypeScript 6 + Angular CLI 22 strictness**: the generated
  `tsconfig.json` no longer carries `"strict": true` explicitly,
  only the additive flags (`noImplicitOverride`,
  `noPropertyAccessFromIndexSignature`, `noImplicitReturns`,
  `noFallthroughCasesInSwitch`). A direct probe ŌĆö a TS file
  with an untyped parameter and a `null` assigned to a
  `string` ŌĆö confirmed that `TS7006` (implicitAny) and
  `TS2322` (strict null) still fire. So strict mode is
  effectively active, just inherited from a newer default
  layer (TS 6 or `@angular/compiler-cli` baseline) rather
  than spelled out. `tsc --showConfig` reflects the
  surface-level config and does not expose this inherited
  state.
- **`@angular/build` is the unified builder** for `build`,
  `serve`, and `unit-test` in Angular CLI 22, replacing the
  earlier mix of webpack-based `@angular-devkit/build-angular`
  builders. esbuild + Vite under the hood.
- **`.vscode/mcp.json`** is a new convention shipped by the
  Angular CLI scaffold. It declares a Model Context Protocol
  server (`npx -y @angular/cli mcp`) that VS Code's
  AI-assistant integrations (Copilot, Continue, ŌĆ”) can invoke
  to get Angular-aware completions and refactors.
- **`.gitignore` precedence and anchoring**: a bare
  `.vscode/` rule at the repository root matches every
  `.vscode/` directory recursively, and per-file
  `!` exceptions in a child `.gitignore` cannot override the
  parent directory being excluded (Git design). The fix is
  to anchor the parent rule with a leading slash
  (`/.vscode/`) so it only matches the root level, leaving
  sub-modules free to manage their own ignore policy.

## Issues encountered & resolved

- **Initial `ng new` failure**: Angular CLI 22 reported *"The
  Angular CLI requires a minimum Node.js version of v22.22.3 or
  v24.15.0 or v26.0.0"*. Resolved by upgrading via nvm to
  Node 24.16.0 LTS and pinning via `.nvmrc`. Shell-state caveat:
  each `Bash` invocation is a fresh shell, so `nvm use` must be
  re-applied (or `PATH` extended manually) in every command ŌĆö
  pin in `.nvmrc` is the long-term safety net.
- **`nvm use` cached PATH**: in one invocation `nvm use` claimed
  to switch to 24.16.0, but `node -v` kept reporting 22.9.0
  because the shell had a stale `hash` cache. Worked around by
  `export PATH="$HOME/.nvm/versions/node/v24.16.0/bin:$PATH"`
  plus `hash -r`.
- **Missing `provideZonelessChangeDetection()` in
  `app.config.ts`** (suspected as an anomaly). Resolved by
  reading the Angular zoneless guide: zoneless is the
  default since v21+, no provider call required. The
  generated `app.config.ts` is correct as-is.
- **Missing `"strict": true` in the generated
  `tsconfig.json`** (suspected as an anomaly). Resolved by
  the TS probe above: strict checks are active. No change
  needed.
- **`.vscode/` files not staged** initially: the root
  `.gitignore` had a bare `.vscode/` rule that overrode the
  whitelist in the local `apps/frontend/.gitignore`. Resolved
  by anchoring the root rule with `/.vscode/`.
