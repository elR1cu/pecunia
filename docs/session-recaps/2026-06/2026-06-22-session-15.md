## Key decisions made
- **Backend-only this session** (frontend SonarCloud deferred): isolates the
  Java 25 scanner risk from the TS/LCOV setup. The Java 25 risk turned out to
  be a non-issue (analyzer fully supports Java 25).
- **CI-based analysis, not Automatic Analysis**: only CI-based can ingest the
  JaCoCo coverage report; Automatic Analysis disabled on the project.
- **Scanner version pinned in the pom + short prefix** (user's call) over an
  unversioned fully-qualified goal in CI — single source of truth,
  reproducible, usable locally.
- **`qualitygate.wait` as a CLI flag, not a pom property**: gate enforcement is
  a CI concern; a local `mvn sonar:sonar` should not block on gate polling.
- **Coverage exclusions limited to framework wiring** (config beans, bootstrap),
  not the masking/logout logic — the latter is security/behavioural and was
  unit-tested instead. Aligns the gate with the testing policy in CLAUDE.md.
- **CSRF finding resolved as a false alarm in context, not "fix later"**: with
  SonarCloud's current binary (Accepted = "fix later" vs False Positive),
  False Positive better fits a dev-only + stateless + read-only chain where no
  CSRF risk materialises.

## Concepts explained / learned
- **`@{argLine}` late substitution** in Surefire is what lets JaCoCo and Mockito
  agents coexist; a hard-coded `argLine` silently discards JaCoCo's → 0%.
- **`sonar.exclusions` vs `sonar.coverage.exclusions`**: the former removes a
  file from *all* analysis (generated code); the latter only from the coverage
  metric (still analysed for bugs/smells).
- **Security Hotspots have been merged into Issues** in current SonarCloud (the
  old To Review → Safe workflow is gone). A "security-sensitive" item is now a
  Security-type Issue resolved as Accepted or False Positive — corrected mid-session.
- **JaCoCo overall % ≠ Sonar coverage %**: JaCoCo ignores Sonar exclusions, so
  its project figure (66%) included the generated DTO and config; Sonar-visible
  coverage was ~85–97%.
- The Sonar scanner runs its analysis on a provisioned JRE (independent of the
  build JDK), so building on JDK 25 is fine.

## Issues encountered & resolved
- **CI red after the first push**: an earlier edit had *replaced* the `mvn verify`
  step with the `sonar:sonar` step, so nothing compiled and the scanner failed.
  Fixed by combining into one `mvn verify sonar:sonar` step (the SonarSource-
  recommended pattern).
- **Splitting one pom across two commits** (JaCoCo vs Sonar): done by restoring
  the HEAD pom, re-applying only the JaCoCo edits + Spotless, committing, then
  restoring the full pom for the Sonar commit.
- **Spotless sortPom** reordered `<goals>`/`<phase>` inside the JaCoCo execution
  (caught by `spotless:check`, fixed by `spotless:apply`).
- **Could not read the Quality Gate status via the public API** (404 — project is
  private on SonarCloud); relied on the user's UI instead.
