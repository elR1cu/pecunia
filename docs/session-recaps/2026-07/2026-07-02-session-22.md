## Key decisions made
- **One port + one service per use case** (Option A) — ISP-strict, testable in
  isolation, showcases hexagonal discipline.
- **Error modeling = C + 2 (parsimonious)** — sealed `Result` only for
  expected, use-case-specific, data-carrying outcomes; everything else is an
  exception. Fine-grained domain exceptions extending `DomainException`, but
  only for domain-only invariants reachable by user input. Captured in ADR-0027.
- **Split `IbanRequiredException` / `IbanForbiddenForTypeException`** rather
  than one exception with a boolean flag (avoids the boolean-parameter smell).
- **`InvalidIbanException` stays no-arg with a fixed message** — a structural
  guarantee that the raw IBAN can never leak into logs (a `String` message ctor
  was tried and reverted).
- **`@Transactional` at method level, at each method's true mode** — writers
  `@Transactional`, reader `readOnly = true`. Rejected the class-level
  `readOnly` default because, on writer services, a forgotten override silently
  suppresses the flush (`FlushMode.MANUAL`) — a silent data-loss trap.
- **Exceptions live in an `exception/` subpackage** at every level (uniform
  convention); `AccountNotFoundException` sits in `application.exception`
  (an application concern per ADR-0027), maps to **404**.
- **`@TenantId` deferred** to the infrastructure slice as its own decision;
  current lean is PostgreSQL RLS over `@TenantId` (RLS enforces below the ORM
  without ambient state). No impact on the ports, which keep `owner` explicit.

## Concepts explained / learned
- **Sealed `Result`**: a closed set of `record` outcomes enabling a
  compiler-checked exhaustive `switch` at the web edge; richer than an enum
  (per-case data), safer than exceptions for expected outcomes.
- **The Result-vs-exception rule (two axes)**: `Result` only if the outcome is
  expected *and* specific to the use case *and* carries data; otherwise an
  exception (unexpected, or expected-but-transverse/data-less).
- **A/C ↔ 1/2 affinity**: fine exceptions (2/3) only pay off if they propagate,
  i.e. under C; pure-A makes them redundant. The dependency is one-way — C also
  works with plain JDK exceptions.
- **`@Transactional(readOnly=true)` semantics**: sets Hibernate `FlushMode` to
  `MANUAL`, so a forgotten writer under a readOnly default loses writes
  silently — hence biasing defaults toward the benign failure mode.
- **`@Valid` vs domain invariants**: DTO validation covers shape/presence at the
  edge; domain-only invariants (IBAN mod-97, cross-field IBAN rules) are the
  domain's own defense-in-depth and are translated to 400 at the web boundary.
