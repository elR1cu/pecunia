## What was done
- Added the `account.domain` package under `apps/api`:
  - `Account` — mutable aggregate root: identity-based `equals`/`hashCode`,
    private canonical constructor, `open` (birth → always `ACTIVE`) and
    `reconstitute` (rehydration) factories, `archive()` (rejects
    double-archive), `balanceFrom(Money)`.
  - `Iban` — value object with normalization (strip whitespace, upper-case)
    and ISO 7064 mod-97 checksum validation.
  - `AccountType` (`requiresIban()`) and `AccountStatus` enums.
  - Enforced the invariant `CREDIT_CARD ⇒ no IBAN`; `CURRENT` / `SAVINGS ⇒
    IBAN required`.
- Author-written unit tests: `AccountTest` (17 cases) and `IbanTest` (9
  cases) — invariants, factories, archive lifecycle, `balanceFrom`,
  identity equality, mod-97 checksum.
- Dropped `allowEmptyShould(true)` on the two `..domain..` ArchUnit rules in
  `HexagonalArchitectureTest` — they now prove the domain is framework-free
  against real classes.
- `refactor(shared)`: added messages to the `Money` null checks.
- Docs: new "Entities and value objects" section in `domain-model.md`
  (mutable aggregates vs immutable value objects, and the rationale), and
  reconciled the `Account` / `Money` descriptions with the implemented code
  (Money scale 4 / HALF_EVEN, `AccountStatus`, `balanceFrom(movementsSum)`,
  `SAVINGS`).
- Merged as PR #20 (squash, signed).

## Key decisions made
- `Account` is modeled as a **mutable class with identity-based equality**
  (aggregate root), not a record — consistent with the entity/value-object
  convention now documented in `domain-model.md`.
- Two factory methods separate the two lifecycles: `open` for birth
  (invariant-enforcing, always `ACTIVE`) vs `reconstitute` for rehydration
  from persistence.
- IBAN validity is enforced in the domain via ISO 7064 mod-97 rather than a
  regex-only shape check.
- The `CREDIT_CARD`-has-no-IBAN rule lives in the domain constructor, keyed
  off `AccountType.requiresIban()`.
