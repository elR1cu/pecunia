## Key decisions made
- **Ports live in the application layer** (Position 2, Hombergs "Clean/Hexagonal"
  style) — domain is the pure model; `application.port.in` (driving) +
  `application.port.out` (driven) + `application.service`. Refines ADR-0003.
  Rationale: one rule with no exception, the simplest ArchUnit invariant,
  consistent with Session 19's `AccountMovements` placement.
- **Input ports kept** (`port.in` use-case interfaces) for full in/out symmetry
  and controller testability.
- **Shared kernel = a package (`com.pecunia.shared`), not a Maven module.**
  ArchUnit enforces the boundary; a separate module would force a multi-module
  reactor/root pom for no gain. JPMS rejected too (only encapsulates *between*
  modules, and Spring Boot on the module path forces self-defeating `opens`).
- **`IdGenerator` as an injected port, not a static factory** — centralizes the
  UUID library in one adapter (relevant for the planned JDK-26 migration) and
  gives deterministic IDs in use-case tests. The interface lives in the kernel
  (a deliberate, documented exception to "ports in application"; it is a
  kernel-level SPI, pure Java).
- **Value objects wrap `java.util.UUID`, not the library's `GUID`** — the
  neutral JDK type stays in the model (native JPA/Jackson/Postgres/OpenAPI
  support); `GUID.v7().toUUID()` is confined to the adapter.
- **`Money` at scale 4 / HALF_EVEN** — guards against rounding drift in derived
  computations (Block 5), covers multi-currency minor units, and fixes
  `BigDecimal` scale-sensitive equality for the record.
- **Cross-cutting technical adapters live in their own top-level packages**
  (`observability`, `security`, `id`) — they would extract to different Maven
  modules if the monolith were ever split, so they are not one unit.
- **`identity` is not converted to hexagonal** — it is a thin BFF adapter over
  the OIDC security context with no domain; hexagonal would be empty ceremony.
- **Coverage**: exclude thin infrastructure adapters (`Uuidv7IdGenerator`),
  unit-test domain value objects rather than excluding them.

## Concepts explained / learned
- **Port placement schools** — classic DDD (repository in domain) vs Clean
  (ports in application); chose the latter for a crisp boundary.
- **JPMS for boundaries** — encapsulates only between modules, so it would force
  one Maven module per context; ArchUnit is more expressive and Spring-friendly.
  The enforcement ladder is ArchUnit → Spring Modulith → (≈ never) JPMS.
- **IdGenerator test determinism** — injecting the generator (like an injected
  `Clock` or seeded `Random`) lets use-case tests assert on a known ID, tracking
  it across return value, persistence and events.
- **`Money` scale 4** — display/settlement precision (2) vs computation precision
  (4); round to the currency's minor units only at the boundary, in HALF_EVEN.
- **Record vs class for an aggregate** — `Account` must be a class with
  identity-based `equals`/`hashCode`; a record's all-fields equality is wrong for
  an entity (two states of the same account would be "unequal").
- **SonarCloud S5778** — in an exception assertion, the lambda must contain only
  the call under test; build fixtures beforehand.

## Issues encountered & resolved
- **Orphan branch after a merged PR**: PR #13 was squash-merged before the JPMS
  commit was added; pushing recreated the (now-closed) branch with an orphan
  commit. Recovered by cherry-picking onto a fresh branch (PR #14) and deleting
  the stale remote/local branch. Lesson: after a squash-merge, new work must
  start from a fresh branch off `main`.
- **ArchUnit rules pointed at a non-existent package**: the baseline targeted
  `com.pecunia.kernel..` while the kernel landed in `com.pecunia.shared` (per the
  ADR). The rules passed vacuously; fixed the target and dropped
  `allowEmptyShould` on the two kernel rules so a green build now *proves* they
  match real classes.
- **Coverage gate failure**: `AccountId`/`UserId`/`Uuidv7IdGenerator` at 0% new-
  code coverage. Resolved by excluding the infra adapter and unit-testing the two
  domain IDs.
- **SonarCloud S5778**: the currency-mismatch assertion's lambda invoked three
  methods; extracted the `Money` fixtures so only `add(...)` remains in the lambda.
