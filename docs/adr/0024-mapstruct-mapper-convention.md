# ADR-0024: MapStruct Mapper Convention

## Status

Accepted

## Context

Pecunia's hexagonal architecture (ADR-0003) keeps three data shapes per
concept:

- **Web DTOs** generated by `openapi-generator-maven-plugin` from
  `contracts/openapi.yaml` (POJOs, see ADR-0023).
- **Domain objects** authored by hand: records, value objects, sealed
  result types. Framework-free.
- **JPA entities** (introduced from Block 2 onwards): annotated classes
  that Hibernate manages.

Crossing these boundaries — DTO ↔ domain on the way in/out of the web
layer, domain ↔ entity on the way in/out of the persistence layer — is a
recurring task. Without convention, each pair grows ad-hoc mapping code:
hand-written static methods, constructors taking neighboring types,
occasional reflection-based libraries. The drawbacks scale with the
codebase:

- **Drift on schema changes**: adding a DTO field forces edits in every
  mapper touching it. A missed edit silently defaults the value to null.
- **Boilerplate weight**: getter/setter pairs are mechanical and
  visually noisy.
- **Runtime cost or fragility**: reflection-based mappers
  (`ModelMapper`, `BeanUtils.copyProperties`) trade compile-time safety
  for runtime surprises that surface late.

The Java ecosystem's de facto answer is **MapStruct**: a compile-time
annotation processor that generates the mapper implementation from an
interface, with no runtime reflection.

### Questions to settle

The project needs to decide more than "use MapStruct or not":

1. How are mappers wired into Spring?
2. What happens when a target field has no source?
3. How are methods and classes named so that mappers are predictable
   across the codebase?
4. How is MapStruct ordered against Lombok in the annotation processor
   chain (Pecunia uses Lombok selectively, ADR-0013)?
5. What escape hatches are accepted for cases MapStruct does not handle
   elegantly (multi-source computation, conditional mapping)?

### Why the convention is worth pinning early

The first mapper (`CurrentUserMapper`, Block 1) is not a flattering
example of MapStruct — four of its five fields need explicit `@Mapping`
because the OIDC source claim names diverge from the DTO field names.
The benefit becomes clear only when mappers proliferate. Setting the
convention before that point avoids inconsistent retroactive
refactoring.

## Decision

Use MapStruct for every mapping that crosses an architectural boundary
(DTO ↔ domain, domain ↔ entity, internal value-object conversions when
the shape change is non-trivial). Pin the conventions below at the
project level.

### Library and build wiring

- Dependency: `org.mapstruct:mapstruct:1.6.3`.
- Annotation processor declared in `maven-compiler-plugin`'s
  `annotationProcessorPaths`. **Lombok is listed first, MapStruct
  second.** Lombok generates accessors before MapStruct introspects the
  source/target types; reversing the order breaks introspection for
  Lombok-annotated classes.
- `lombok-mapstruct-binding` is **not** added until a Lombok-annotated
  class actually flows through a MapStruct mapper. YAGNI: the binding
  resolves an interaction that does not exist at Block 1 (DTOs are POJOs
  from codegen, the domain layer forbids Lombok per ADR-0013).

### Mapper interface declaration

Every mapper is an `interface` annotated with:

```java
@Mapper(
    componentModel = "spring",
    unmappedTargetPolicy = ReportingPolicy.ERROR
)
```

- `componentModel = "spring"`: the generated `*Impl` class is a
  `@Component`, eligible for constructor injection. Mappers are
  injected like any other Spring bean; no manual instantiation, no
  static factory.
- `unmappedTargetPolicy = ReportingPolicy.ERROR`: adding a target field
  without the matching `@Mapping` directive fails the Maven build. The
  default `IGNORE` policy silently defaults the field to null at
  runtime; for a finance application this is unacceptable.

### Naming

- **Class name** carries the **concept** being mapped, suffixed
  `Mapper`. Examples: `CurrentUserMapper`, `TransactionMapper`,
  `AccountMapper`. The class name is **not** parameterized by direction.
- **Method name** carries the **direction**:

| Method | Source | Destination | Used at |
|---|---|---|---|
| `toDto` | domain (or external type) | web DTO | web layer outbound |
| `toEntity` | domain | JPA entity | persistence layer outbound |
| `toDomain` | DTO or entity | domain | web/persistence inbound |

A single mapper interface may declare multiple methods covering the
relevant directions for its concept:

```java
@Mapper(componentModel = "spring", unmappedTargetPolicy = ReportingPolicy.ERROR)
public interface TransactionMapper {
    Transaction toDomain(TransactionDto dto);
    TransactionDto toDto(Transaction transaction);
    TransactionEntity toEntity(Transaction transaction);
    Transaction toDomain(TransactionEntity entity);
}
```

Overloads on the same method name are unambiguous to MapStruct as long
as parameter types differ.

### Multi-source field computation

MapStruct's `@Mapping` directive handles single-source field mapping
elegantly. Multi-source computation (e.g., a DTO field derived from two
source claims) is awkward:

- `@AfterMapping` mutates the target after construction, which only
  works when the target has setters (not a record).
- Declaring two parameters and using `source = "<paramName>"` becomes
  ambiguous when multiple sources expose the same field name.
- A custom default method on the interface, called via
  `expression = "java(...)"`, is verbose but isolates the logic.

The escape hatch chosen for the project: `expression = "java(...)"`,
calling a `default` method declared on the mapper interface. This keeps
the computed logic close to the mapping declaration without polluting
the generated `*Impl` class.

```java
@Mapping(target = "displayName", expression = "java(resolveDisplayName(oidcUser))")
CurrentUser toDto(OidcUser oidcUser);

default String resolveDisplayName(OidcUser user) {
    return user.getFullName() != null ? user.getFullName() : user.getPreferredUsername();
}
```

### Tests

Slice tests that exercise a controller using a mapper load the real
`*Impl` via `@Import(CurrentUserMapperImpl.class)` rather than mocking
the mapper. The controller-to-mapper integration is part of the
contract being tested, and the generated `*Impl` is fast (no
reflection, no I/O).

## Consequences

### Positive

- **Drift caught at build**. A new target field without its `@Mapping`
  fails compilation; a removed source field surfaces as a compile error
  in the generated `*Impl`.
- **Zero runtime reflection**. Method calls in the generated `*Impl`
  are plain Java, fully optimizable by the JIT and visible in stack
  traces.
- **Spring-native wiring**. The generated `*Impl` is a `@Component`;
  constructor injection from a controller or service is identical to
  any other bean.
- **Convention is uniform**. A new mapper file is predictable: a
  reader knows the annotation, the policies, the method naming, and the
  injection point without reading the class.
- **Test integration is mechanical**. `@Import({SecurityConfig.class,
  CurrentUserMapperImpl.class})` in a `@WebMvcTest` slice loads the
  real mapper alongside the controller.

### Negative

- **Generated code is opaque** to the file the developer edits. Reading
  the actual mapping logic requires navigating to
  `target/generated-sources/annotations/.../*MapperImpl.java`. Most
  IDEs handle this transparently; developers new to MapStruct may be
  surprised.
- **Multi-source computation needs an escape hatch.** The
  `expression = "java(...)"` pattern is verbose and string-typed
  (compile errors in the expression body surface as cryptic messages in
  the generator output).
- **Generated `*Impl` uses no-args constructor + setters**, not the
  record's canonical constructor. This means Jakarta Bean Validation
  annotations on record components are not enforced at construction
  time. For records used as targets, validation must happen elsewhere
  (typically at the boundary, via `@Valid` on the controller's
  parameter).

### Neutral

- **First-time learning curve.** A developer encountering MapStruct
  for the first time must read the basics. This is a one-time cost.
- **Annotation processor ordering matters silently.** Lombok before
  MapStruct works; the reverse fails silently for Lombok-annotated
  sources. The order is pinned in `pom.xml`; a future contributor
  reordering for "alphabetical cleanliness" would break things.
  Mitigated by an inline comment in the pom.

## Alternatives Considered

### Hand-written mappers

Static methods that copy fields manually.

Rejected because:

- Maintenance grows linearly with the number of field pairs. Each
  schema change touches multiple files.
- No compile-time check that target fields are covered. A missed
  copy silently produces a default value at runtime.
- Boilerplate density makes code review fatiguing.

Acceptable as a one-off for a trivial conversion (one or two fields,
unlikely to grow). Not as a project-wide convention.

### ModelMapper (runtime reflection)

A reflection-based mapping library; mappers are configured at runtime
through fluent builders or convention-based name matching.

Rejected because:

- No compile-time safety. Mismatches surface as runtime exceptions or
  silent nulls.
- Reflection cost on every mapping call. Not free, especially in a
  budgeting app that processes thousands of transactions on import.
- Stack traces are obscured by ModelMapper internals.

### MapStruct without `unmappedTargetPolicy = ERROR`

The library's default is `ReportingPolicy.IGNORE`: unmapped target
fields are silently set to null.

Rejected because:

- Adding a field to a DTO without updating the mapper would set that
  field to null at runtime. For monetary values, account references,
  or user identifiers in a finance application, this is a defect class
  worth eliminating mechanically.
- The build-time noise from explicit `@Mapping` declarations or
  `@Mapping(ignore = true)` is a feature, not a cost: it forces the
  developer to consciously address every new field.

### `componentModel = "default"` with manual instantiation

The generated `*Impl` is a plain class; consumers `new XxxMapperImpl()`
or expose a static factory.

Rejected because:

- The codebase is already Spring-managed. Mixing manually-instantiated
  components with DI-managed ones creates an inconsistent wiring story.
- Tests would need to instantiate the mapper themselves instead of
  receiving it via context.

### Lombok `@Builder` + manual factory methods

Use Lombok to reduce boilerplate on hand-written builders.

Rejected because:

- Still manual: the drift problem remains.
- Lombok in the domain layer is forbidden by ADR-0013, so this only
  applies to DTOs and entities — half the use case.

### `@AfterMapping` for multi-source fields

Mutate the target after MapStruct has built it from the primary source.

Rejected for record targets (no setters). Even for non-record targets,
the mutation is harder to follow than an inline `expression`: the
mapping declaration no longer tells the full story, the reader has to
search for the `@AfterMapping` method.

The `expression = "java(...)"` escape hatch keeps everything in one
place at the cost of string-typed code.

### One mapper per direction (e.g., `CurrentUserToDtoMapper`)

Split each direction into its own interface.

Rejected because:

- File count doubles for each concept that needs both directions.
- The concept is the same; only the direction differs. A single
  interface with two methods groups the related logic.
